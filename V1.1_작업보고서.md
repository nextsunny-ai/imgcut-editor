### JS syntax 검증
- 내장 `<script>` 블록 2개 (= 총 139,834 chars) → Node v25.8.1 `--check` 통과 (exit 0)
- 어제 사고 (`let currentLang` TDZ + 어퍼스트로피 `Don't` 사고) 회피 = i18n 메시지는 어퍼스트로피 대신 `—` 사용. `let currentPageSize` 선언 = 새 함수 블록 안 한 군데만.

- **육안 디자인 검증 (= 캡처된 슬라이드가 깨지지 않았는지)** — magic header + size만 검증. 실제 슬라이드 레이아웃이 정상인지는 = 메인 클로드 + 대표님이 PowerPoint 열어서 확인.

## 3. 알려진 한계 (= 사용자에게 toast로 안내됨)

### 3-1. PPT 슬라이드 = 이미지 = 텍스트 편집 X
- `html2canvas`로 페이지를 canvas 캡처 → `slide.addImage()`로 PPT 슬라이드에 박음.
- 따라서 PowerPoint에서 슬라이드 = 그림 1장. 텍스트 박스 선택·편집 불가.
- 사용자 안내 = `'msg.pptReady'` 키 = "PPT 다운로드 완료 (슬라이드는 이미지 — 텍스트 편집 안 됨)"  / "PPT downloaded (slides are images, text not editable)"
- 진짜 텍스트 편집형 PPT = HTML 구조 분석 + pptxgenjs `slide.addText()` 호출 + 폰트/위치 변환 = 큰 별도 작업.

### 3-2. 페이지 사이즈 = body 강제 width 적용 방식
- `applyPageSizeToHtml` = `<style id="sunny-pagesize-override">`에 `html, body { width: 1920px !important; ... }` 박음.
- 사용자 HTML이 이미 `width: 100vw`·flex·grid 기반이면 = 어색하게 보일 수 있음. 페이지 사이즈 = "PDF·PPT 캡처용 비율 강제" 용도. UI 디자인 자체를 새로 짜는 게 아님.
- 미리보기 iframe에도 적용되므로 = 사용자가 미리 확인 가능.

### 3-3. 캡처 비율 fit 방식
- canvas 비율과 PDF/PPT 슬라이드 비율이 다르면 = letterbox (= 위·아래 또는 좌·우 흰 여백) 발생. 강제 잘라내기 X = 콘텐츠 보존 우선.

### 3-4. CDN 의존성
- 인터넷 연결 필요 (= cdnjs + jsdelivr). offline 사용 X.
- 추후 PWA offline 모드 박을 때 = `sw.js`에 3개 CDN URL 캐싱 박는 게 권장.

### 3-5. 큰 사이트 = 느림 가능
- html2canvas는 페이지 크면 = 캡처에 5~30초 소요 가능. 페이지마다 5초 timeout safety 박힘 (`captureHtmlToCanvas` 내부).
- 진행 오버레이 (`prog.pdf`·`prog.ppt`·`prog.pageCapture`) 박혀서 사용자에게 "X 페이지 캡처 중 N/총" 표시.
